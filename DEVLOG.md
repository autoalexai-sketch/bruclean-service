# DevLog — brucleanservice.pl

## 2026-07-22 — Баг мобильного меню («вместо меню высвечивается номер телефона»)

### Симптом
На реальном телефоне и в инкогнито-вкладке при открытии мобильного меню (гамбургер) вместо полноэкранного меню появлялся оранжевый номер телефона поверх лого в шапке. Меню фактически не открывалось.

### Диагностика
1. Первая гипотеза — устаревший кэш CSS. Добавлен cache-busting (`?v=20260722`) ко всем стилям на всех 11 страницах. **Не помогло** — баг воспроизвёлся в свежей инкогнито-вкладке, кэш исключён.
2. Диагностика через DevTools пользователя (Device Toolbar, 400×568):
   - `getComputedStyle` подтвердил, что десктопная плашка с номером телефона корректно скрыта (`display:none`) на мобильном разрешении — это был не тот элемент.
   - `document.elementFromPoint(100, 40)` показал, что на этом месте реально рендерится `#mobile-menu` (открытое состояние, без класса `hidden`), но без эффекта позиционирования.

### Причина №1 — отсутствующие CSS-правила
`css/tailwind.min.css` — это **статическая, вручную собранная сборка Tailwind**, а не живой компилятор. Более ранний фикс мобильного меню (full-screen + scroll-lock) добавил в HTML классы `inset-x-0`, `top-20`, `overscroll-contain`, но соответствующие правила в CSS-файл добавлены не были — классы молча не работали. Из-за отсутствия `top-20`/`inset-x-0` `position:fixed`-блок меню не имел позиционирования и рендерился в случайном месте у шапки.

**Важный вывод:** любой новый Tailwind-класс, добавленный в HTML этого проекта, нужно вручную добавлять в `css/tailwind.min.css`, иначе он не будет работать — компилятор здесь не запускается автоматически.

Фикс: добавлены правила `.inset-x-0{left:0;right:0}`, `.top-20{top:5rem}`, `.overscroll-contain{overscroll-behavior:contain}`.
Коммит: `eabd949` — *Add missing utility classes (inset-x-0, top-20, overscroll-contain)*.

### Причина №2 — конфликт z-index с cookie-баннером
После фикса №1 меню стало открываться и позиционироваться правильно, но у первых посетителей (первая загрузка / инкогнито) поверх меню оставался виден только первый пункт «Oferta» — всё остальное перекрывал баннер cookie-согласия.

Причина: `#mobile-menu` имел `z-40`, а `#cookie-banner` (`fixed bottom-0 left-0 right-0`) — `z-50`. Баннер визуально лежал поверх открытого меню.

Фикс: z-index меню поднят до `z-[60]` (выше баннера) во всех 8 файлах, где есть `#mobile-menu` (index.html + 7 страниц `uslugi/*`), плюс добавлено CSS-правило `.z-\[60\]{z-index:60}`.
Коммит: `ebeca03` — *Fix mobile menu z-index so it renders above the cookie consent banner*.

### Причина №3 — backdrop-filter шапки ломал position:fixed
После фиксов №1 и №2 пользователь сообщил: меню стало «со скроллом и очень маленькое». Живой CSS-файл на сайте при проверке моими инструментами продолжал казаться «устаревшим» несколько часов подряд — расследование через GitHub API (`/deployments`, `/deployments/{id}/statuses`) показало, что GitHub Pages исправно задеплоил все коммиты (`state: success`) практически сразу после пуша. Значит проблема была не в деплое и не в CDN-кэше, а в моём собственном инструменте проверки (`web_fetch`), который стабильно отдавал устаревшую копию файла — важный урок: не доверять одному каналу проверки, если он единственный показывает расхождение с независимо подтверждённым состоянием (GitHub API).

Реальная причина: `<div id="mobile-menu">` физически находился **внутри** `<header>`, а `<header class="... backdrop-blur-md ...">` — то есть с CSS `backdrop-filter`. По спецификации CSS, `backdrop-filter` (как и `filter`/`transform`/`perspective`) у предка создаёт новый containing block для потомков с `position:fixed`. Из-за этого `top-20`/`inset-x-0`/`bottom-0` меню считались не от вьюпорта, а от ~80px-высоты самой шапки — меню схлопывалось в крошечный блок с внутренним скроллом.

Фикс: `<div id="mobile-menu">` вынесен из `<header>` наружу, сразу после `</header>`, без изменения содержимого/классов. Сделано во всех 8 файлах.
Коммиты: `56dd4ba` (index.html) и по одному на каждый файл `uslugi/*.html`, последний — `b09f4fc`.

### Favicon (22–23.07.2026)
На сайте отсутствовал favicon (не было ни файла, ни `<link rel="icon">`). Вырезан квадратный кроп логотипа-домика из `image_34e979.png`, сгенерирован PNG 64×64.

**Важный технический нюанс:** GitHub MCP-инструменты `create_or_update_file`/`push_files` в этой среде трактуют параметр `content` как обычный UTF-8 текст и не декодируют base64 — попытка запушить бинарный `.ico`/`.png` файл напрямую (даже в виде base64-строки) сохраняет в репозитории буквально текст base64, а не картинку. Решение: favicon встроен **inline как data-URI** прямо в `<link rel="icon" type="image/png" href="data:image/png;base64,...">` в `<head>` каждой из 11 страниц — отдельных бинарных файлов в репозитории нет и не нужно.
Коммит: `23ab2dc` (index.html) + по одному на остальные 10 страниц.

### Инфраструктура (контекст)
Хостинг — GitHub Pages за Fastly CDN (не Cloudflare; подтверждено заголовками `server: GitHub.com`, `via: 1.1 varnish`, `x-served-by`). Каждая edge-точка Fastly кэширует HTML/ассеты до ~10 минут (`cache-control: max-age=600`) — после пуша разным пользователям в разных регионах может на короткое время отдаваться разный контент.

### На будущее
Стоит перепроверить, не вносили ли другие недавние правки (Meta Pixel, cache-busting, UI/UX-аудит) новые Tailwind-классы, отсутствующие в `tailwind.min.css` — риск такой же silent-fail ошибки сохраняется при любой будущей правке HTML без синхронного обновления статического CSS.
